하루가 끝나면 뭘 했는지 기억이 안 납니다.

정확히 말하면, 뭘 했는지는 어렴풋이 기억나는데 **어떤 기분이었는지**와 뒤섞여 있습니다. "오늘 서버 설정 때문에 3시간 날렸다"는 사실과 "왜 이것도 못 하나 싶어서 짜증났다"는 감정이 한 덩어리로 기억됩니다. 그래서 돌아보면 "오늘 힘들었다"만 남습니다. 뭘 했는지는 잊고, 힘들었다는 감정만 누적됩니다.

이게 반복되면 번아웃 루프에 빠집니다. 실제로 많은 일을 했는데, 기억에는 피로만 남으니까 "나는 아무것도 못 했다"고 느낍니다. 첫 편에서 말한 **결핍-충족 루프**의 연료가 바로 이 왜곡된 기억입니다.

그래서 기록 시스템을 만들었습니다. 핵심은 단순합니다: **감정과 사실을 분리합니다.**

## Before: 뒤섞인 기록

올해 초, 기록을 시작하던 시기의 업무 일지는 이런 형태였습니다:

```
- 신규 기능 기획 논의 내용 정리
- 문서 요청 범위 해석 차이 발생 → 롤백 후 별도 문서 생성
- 동료 강의 메모를 보며 실행 밀도에 대한 압박감을 다시 느꼈다
- 오늘은 정리의 질은 높였지만 핵심 지표에 직접 닿는 실험까지는 못 갔다
- 원하지 않는 fallback 로직이 계속 끼어들어 답답했다
- 내가 만든 결과가 기존 방식보다 나은지 판단이 안 되는 상태에서
  초안을 계속 고치려니 더 깜깜하게 느껴졌다
```

사실과 감정이 같은 목록에 나열되어 있습니다. 다음 날 이걸 다시 읽으면, "답답했다"와 "깜깜하게 느껴졌다"가 전체 톤을 지배합니다. 실제로는 문서를 2트랙으로 분리하고, 검증 플로우를 정리하고, PR도 올린 생산적인 날이었는데.

반대로, 이런 날도 있었습니다:

```
## 오늘 한 일
- main 브랜치 merge 및 rollback 커밋
- CS 대응: DB 토큰 조회 시도 → 권한 정책에 차단됨
- 우회 접근 시도 (여러 경로) → 모두 실패

## 회고

```

회고가 비어 있습니다. 감정을 아예 안 쓴 날. 이런 날은 기록은 있지만 회고가 없으니, 나중에 "이 시기에 어떤 상태였는지"를 알 수 없습니다.

**문제는 두 가지입니다**: 감정이 사실을 오염시키거나, 감정이 아예 기록되지 않거나.

## 3단계 파이프라인

이 문제를 풀기 위해 만든 시스템이 3단계로 동작합니다.

### 1단계: 세션 마무리 — `/session-log`

하나의 작업 세션(보통 2~3시간)이 끝나면, `/session-log`를 실행합니다. AI가 이번 세션에서 뭘 했는지 요약해서 **Daily Buffer**라는 파일에 추가합니다.

여기서 중요한 건, Buffer에는 뭐든 써도 된다는 것입니다. "DB 접근이 막혀서 3시간 날림"도 쓰고, "삽질이 너무 길어서 자괴감 들었다"도 씁니다. 분류는 나중에 합니다. 이 단계의 목표는 **유실 방지**입니다.

### 2단계: 하루 마무리 — `/daily-log`

하루가 끝나면 `/daily-log`를 실행합니다. 이때 AI가 세 가지 소스를 읽습니다:

1. **Daily Buffer** — 오늘 세션들에서 누적된 메모
2. **Git log** — 오늘 커밋한 내역
3. **이슈 트래커** — 커밋 메시지에서 추출한 티켓 정보

### 3단계: 감정-사실 분류

여기가 핵심입니다. AI가 Daily Buffer의 각 항목을 네 가지로 분류합니다:

| 유형 | 기준 | 가는 곳 |
|------|------|---------|
| **사실** | 뭘 했고, 뭐가 막혔고, 어떻게 풀었는지 | `오늘 한 일` / `이슈 및 해결` |
| **감정** | 뭐가 불안하고, 뭐에 뿌듯했는지 | `회고` |
| **아이디어** | 갑자기 떠오른 구상이나 개선안 | `00-Inbox/`에 개별 노트 |
| **액션** | 다음에 해야 할 일 | `내일 할 일` |

하나의 문장에 사실과 감정이 섞여 있으면 분리합니다:

```
"데이터 파이프라인 배포 완료했는데 좀 뿌듯하다"
→ 사실: "데이터 파이프라인 배포 완료"
→ 감정: "뿌듯함"
```

분류가 끝나면 사용자(나)에게 보여주고, 수정할 기회를 줍니다. 그런 다음 각 항목이 데일리 노트의 해당 섹션으로 자동 배치됩니다.

감정이 아예 없는 날도 있습니다. Buffer에도 세션 버퍼에도 감정 표현이 하나도 없으면, AI가 짧은 회고 인터뷰를 합니다:

> "오늘 작업 중 가장 에너지를 많이 쓴 건 뭐였어요?"
> "지금 컨디션은 어때요?"
> "오늘 하루를 한 단어로 표현하면?"

짧게 답하면 그걸로 회고 2~3줄을 써줍니다. "패스"하면 비워둡니다.

## After: 분리된 기록

같은 날의 기록이 이렇게 바뀝니다:

```
## 오늘 한 일
### 검색 분석 고도화
- 검색 상태 클라우드 업로드 기능 구현 (PR 올림)
- 포털 데이터 수집 실패 수정 (PR 올림)

### 리서치 데이터 수집
- A 사이트 통계: 41개 통계 803행
- B 사이트 진료정보: iframe JSON API 발견
- C 사이트: 봇 차단 우회, 304건 수집

## 이슈 및 해결
- C 사이트 봇 차단 → navigator.webdriver 숨기기로 우회
- C 사이트 필터 조건 오류 → 파라미터 코드가 예상과 달랐음
- B 사이트 빈 응답 → iframe 내 별도 JSON API 발견

## 회고
크롤러 7개를 하루에 돌리면서 각기 다른 봇 차단 패턴을 뚫는 게
게임 같아서 재밌었다. 파라미터 코드가 예상과 달랐던 건 진짜 웃겼다.
숨겨진 API를 발견했을 때 뿌듯했다.
양이 많았지만 소진보다는 성취감이 큰 날.
```

**사실 섹션**은 PR 번호, 수치, 파일 경로 같은 검증 가능한 정보만 남습니다. **회고 섹션**은 감정만 모여서 "이 날 나의 상태"가 명확하게 보입니다.

## 분기 회고에서의 효과

이 시스템으로 1Q 분기 회고를 쓸 때, 극적인 차이가 있었습니다.

3주치 데일리 로그의 회고 섹션만 모아 읽으니, 감정의 궤적이 보였습니다:

- W10 초반: "실행 밀도에 대한 압박감", "깜깜하게 느껴졌다"
- W10 후반: "드라이브를 거니까 상대도 따라오는 느낌이 좋았다"
- W11: "폰에서 노트를 바로 수정할 수 있는 환경이 갖춰지니 안도감"

감정이 별도 섹션으로 분리되어 있으니까, 이걸 연결해서 읽는 것만으로 "1Q에 나는 어떤 상태였는가"가 자명해졌습니다. 1Q 회고에서 이렇게 쓸 수 있었습니다:

> "FOMO는 루프다. 불안 → 과투입 → 단기 안도 → 기준선 상승 → 더 큰 불안. 이 구조를 인식했으니 2Q에는 빠지지 말자."

이 한 줄은, 감정 데이터가 분리되어 누적되지 않았으면 나올 수 없었습니다.

## 기술적으로 어떻게 돌아가는가

특별한 인프라는 없습니다. Claude Code의 **슬래시 커맨드(스킬)**로 구현되어 있습니다.

- `/session-log`: 현재 세션 요약 → Daily Buffer에 append
- `/daily-log`: Buffer + git log + Jira → 분류 → 데일리 노트 생성

스킬은 마크다운 파일 하나에 프롬프트로 정의되어 있고, Claude Code가 실행하면 Obsidian vault의 파일을 직접 읽고 씁니다. 서버도, 크론잡도, 별도 앱도 없습니다.

분류 정확도는 체감상 90% 이상입니다. 가끔 "이건 사실이 아니라 감정인데"라는 오분류가 있지만, 분류 결과를 보여주고 수정할 기회를 주기 때문에 큰 문제는 되지 않습니다. 오히려 **AI의 분류를 검토하는 과정 자체가 하루를 돌아보는 회고 행위**가 됩니다.

## 왜 이게 번아웃 방지에 도움이 되는가

첫 편에서 결핍-충족 루프를 말했습니다. 그 루프의 연료는 **왜곡된 자기 인식**입니다. "나는 아무것도 못 했다", "나만 뒤처진다", "오늘도 허비했다". 이런 생각은 대부분 감정이 사실을 덮어씌운 결과입니다.

감정과 사실이 분리되면:

1. **사실만 보면** — 실제로 많은 일을 했다는 걸 확인할 수 있습니다
2. **감정만 보면** — 불안의 패턴이 보입니다 (언제, 어떤 트리거로 발생하는지)
3. **둘을 교차하면** — "생산적인 날인데 불안했다"와 "비생산적인 날인데 평온했다"를 구분할 수 있습니다

이 구분이 가능해지면, **"쉬어도 불안하다"의 실체**가 드러납니다. 불안의 대상이 구체적으로 보이면, 대응도 구체적이 됩니다. 막연한 공포는 관리할 수 없지만, "화요일마다 피어 비교 후 FOMO가 올라온다"는 패턴은 관리할 수 있습니다.

관찰할 수 있는 것은 관리할 수 있습니다.

## 자주 묻는 질문

**Q. AI 업무일지 자동화에 코딩 실력이 필요한가요?**
A. 프롬프트를 마크다운 파일에 작성하는 수준입니다. Claude Code의 슬래시 커맨드(스킬)를 활용하므로 별도 코딩은 필요 없습니다.

**Q. Obsidian이 아닌 다른 노트 앱에서도 가능한가요?**
A. 핵심은 "마크다운 파일을 직접 읽고 쓸 수 있는 구조"입니다. Logseq, Dendron 등 로컬 마크다운 기반 앱이면 같은 접근이 가능합니다.

**Q. 감정-사실 분류를 AI가 틀리면 어떻게 하나요?**
A. 분류 결과를 먼저 보여주고 수정할 기회를 줍니다. 오히려 AI의 분류를 검토하는 과정 자체가 하루를 돌아보는 회고 행위가 됩니다.
